# Verified Candidates v2: External Dependency Formation Research

## 1. Executive summary

This document lists public, source-linked repositories where AI-generated pull requests, coding-agent commits, or agent-authored automation are already visible, and where ContinuityOS StateGate / Agent Attribution could plausibly address a concrete workflow problem.

Scope discipline:

- **Public evidence only.** No private claims, no outreach, no dependency proof.
- **Verified facts are separated from inference.** Each candidate has public source links; adoption likelihood is an inference, not a claim of interest.
- **No contact was made.** Contact paths are public repo channels only.
- **Candidate rationale.** A repository is included when there is evidence of agent/AI authorship or AI-review automation plus visible PR/review/merge workflow where provenance, merge legitimacy, or attribution enforcement could matter.

Overall findings:

- AI attribution is inconsistent: some repos preserve explicit trailers such as `Generated with Claude Code` / `Co-Authored-By`, while others publish instructions or scripts to remove or suppress AI attribution.
- Workflow pain is visible in both directions: maintainers want transparent AI provenance, but contributors also object to involuntary co-author trailers and contributor-graph pollution.
- The strongest first-contact targets are repos that already expose PR automation, review bots, or explicit attribution cleanup controls; these have the clearest fit for a neutral guardrail that validates the merge object and records agent involvement without over-claiming authorship.

## 3. Top 5 first-contact targets

1. **module-federation/core** — Highest fit because the repo already contains a Claude attribution cleanup script, meaning the pain is concrete and repo-local. A neutral StateGate could preserve provenance without forcing unwanted co-author contribution semantics.
2. **enyst/OpenHands-Tab** — Multiple AI bots appear on PR/action surfaces, including code-assist and review agents. This is a strong live example of multi-agent PR provenance needing normalization.
3. **OpenHands/benchmarks** — Agent-generated issues and closing PRs exist in benchmark infrastructure. Exact-object validation and agent attribution map closely to benchmark/reproducibility needs.
4. **Klipper3d/klipper** — Maintainers publicly discussed AI-assisted code guidelines in a safety-adjacent firmware project. A merge guard could turn ambiguous disclosure norms into deterministic review evidence.
5. **flyteorg/flyte** — Large infrastructure project with public evidence of Claude-generated PR content. StateGate could improve reviewer trust for agent-assisted changes in complex distributed systems.

## 5. Recommended next outreach step

Do **not** contact maintainers yet. First prepare a one-page, non-invasive proposal and a demo branch that can be evaluated without installing runtime infrastructure:

1. Define the exact GitHub signal ContinuityOS would check: PR head SHA, merge base, required status checks, agent attribution trailer/body marker, reviewer approval state, and generated proof object.
2. Provide two opt-in modes:
   - **Attribution-preserving mode:** records agent/tool/model involvement without adding a GitHub co-author.
   - **Attribution-enforcing mode:** requires a declared AI-assistance marker before merge.
3. Create a small public demo against a toy repo showing: unsigned AI PR, attributed AI PR, rewritten attribution, and validated merge object.
4. After demo proof exists, open low-pressure issues or discussions only on the top 2-3 targets, framed as research feedback rather than adoption solicitation.
