# Cooldown Dependency Work Queue

Bounded work queue for keeping Codex productive during cooldown without expanding
ContinuityOS architecture, canon, ontology, or execution semantics.

## Compression

```text
Architecture Proof     COMPLETE
Demonstration Proof    COMPLETE
Dependency Proof       OPEN
```

The active bottleneck is **external dependency formation**: one outside maintainer
must install the agent-attribution gate, make it load-bearing, observe a real
`NULL`/`VALID` boundary, and report that removal would make their merge path worse.

## Operating constraint

Keep every cooldown task anchored to:

```text
one workflow
one external user
one dependency hypothesis
one observable proof
```

If a task requires new primitives, new canon layers, new ontology, new governance
theory, or broad runtime mutation, it is out of scope for cooldown dependency work.

## P0 — Dependency formation

| Work item | Codex-safe output | Human boundary | Proof artifact |
|---|---|---|---|
| External maintainer candidate research | Read-only candidate table updates: active repo, AI-agent usage signal, recent activity signal, outreach readiness | Do not contact maintainers or infer intent | Updated candidate row with dated public evidence links |
| Installability friction audit | Fresh-clone transcript following public docs exactly; every failure point recorded without fixing unrelated systems | Do not claim external install | `npm install` / demo transcript plus failure log |
| Sandbox compression | Single under-5-minute path covering `VALID`, `NULL`, and proof artifact observation | Do not add new architecture to make the demo impressive | Timed transcript with exact commands and observed verdicts |
| Agent wedge demonstration | One agent-lane action showing `VALID | NULL` and proof artifact for a single action only | Do not automate outreach or fabricate external maintainer behavior | PR/check-run link or local transcript showing the boundary |

## P1 — Enforcement / evidence maintenance

| Work item | Codex-safe output | Guardrail |
|---|---|---|
| Attribution conformance tests | Narrow test additions for existing attribution classifications | No new policy semantics unless explicitly scoped |
| Dependency tracker maintenance | Status-only updates when evidence exists | Leave outside-owner rows empty until a real install exists |
| StateGate documentation consistency | Copy/path/version consistency patches | Do not change runtime guard behavior |
| Standing-authority operational analysis | Read-only analysis of existing authority surfaces | No authority widening or execution-path mutation |

## Avoid during cooldown

- New ontology.
- New canon layers.
- New distributed legitimacy theories.
- New cognition-governance expansion.
- New primitives.
- Broad refactors or runtime rewrites.

## NULL criteria

Return planning-only / `NULL` instead of implementation when the requested task:

- Depends on an unverified external maintainer intention.
- Requires contacting a maintainer from Codex.
- Requires changing execution semantics to improve the demo.
- Cannot produce a deterministic evidence artifact.
- Expands beyond the current dependency hypothesis.

## Evidence handoff checklist

Before marking any cooldown item complete, record:

1. The exact workflow under test.
2. The external user or candidate boundary being evaluated.
3. The dependency hypothesis being tested.
4. The observable proof artifact produced.
5. Any unresolved drift or reconciliation gap.
